Google CodeLabs Kotlin Part One

Posted on 2020-09-13 In Google , Kotlin , CodeLabs Facebook: 0

Today I want to talk about the google code labs part one. I have decided to start these to pursue a google android associate developer certificate. This will serve as a reference for things I felt were very important throughout the Google Codelabs series. There are a few important takeaways from this section. The biggest I think was how to use the FindViewById efficiently and not putting it in an onClick method. It is best to declare the button at the top level of the class as a lateinit var, and then FindViewById in the onCreate. The less times you use the FindViewById call the better, as it is a very expensive operation. Each time you call FindViewById, the android system searches the entire view hierarchy.

-Activities
First MainActivity is a subclass of AppCompatActivity. Activities in a sense are what you see when you load into an app. Activities are a base class for a screen. They are usually called an activity, because each screen represents one thing that you are trying to do. The setContectView() method in MainActivity grabs the xml file and inflates it when created. When I say inflate, it is a process where the views defined by xml are turned into Kotlin view objects in memory. Upon inflation the activity can call these objects to the screen and modify them.

-Views
All UI elements in the layout are subclasses of the View class. For example TextView, Button, EditText are examples of views. View elements can grouped inside a ViewGroup. A view group acts as a holder for the views, or other view groups. For example a LinearLayout nested inside a RelativeLayout inside of another layout. You have to reference a view object to connect the kotlin code you typed and the layout you made once it’s been inflated. You do so by assigning an ID to your view, and using the FindViewById call to then find it. You can also code the attributes for a view in Kotlin, such as setting the text, etc. Lastly the setOnClickListener() method is used to attach a click handler to a view. Click handlers are called when the user taps on a UI element such as a button.

-Strings
You should never hard code your strings in an application. They should all be contained in the Strings resource file.

-Toasts
A toast is a view that shows the user a simple message in a pop up on the screen. You call the makeText() method on a Toast class with four arguments. First is the context of the app, second is the message, and third is the length of how long it should appear. Lastly make sure to call the .show() method on it to display it.

-Vector drawables VS Bitmap images
Vector drawables are much better in terms of scaling on different devices without losing quality. They are also better because typically the file size is smaller with the same image versus a bitmap image. Vector drawables is supported from API 21 and up.

-tools:src=”@drawable/dice_1”
This allows you to view your image in the XML design tab. This is very useful if your wanting to preview how the app should look. It will not be loaded on runtime though.

-AndroidX
The app:srcCompat attribute uses the Android X library to support vector drawables in older versions of Android, all the way back to API level 7. So even though my minimum SDK is 19, it allows me to use vector drawables back to API level 7.

This is my first post, so I figured it would be good to do an introduction. Hopefully you could tell that my name is Devin Annunzio. I have about 21 credits left for a Bachelor of Arts in Asian Studies that’s focused on international business. During my time at university, I interned as an Android developer at Possible Mobile, as well as had a very interesting capstone project working with CribMaster. My dream is to work as an Android Developer at Line, or another company in Japan in general.

My hobbies outside of programming are bboying, skateboarding, studying Japanese, and the gym. I guess I’ll tell you about how I got into programming. I originally didn’t know what I wanted to do as a job after graduating high school. So I figured I would go into accounting, as I had started working as an accounting assistant at Harrison Design in Atlanta. As time went on, I wondered if accounting was really for me. I couldn’t see myself working as an accountant for the rest of my life. I ended up moving out to Seattle for about two years, for a change of pace. So why did I create this blog you might be asking. I created this blog for a couple reasons. The reasons are to serve as a tool to track my progress as a developer, to serve as a reference in the future, and hopefully to help other programmers who may struggle with the same things I did. I don’t believe I am extremely smart or gifted. So I hope that this can inspire others to feel that anything is possible, and that they too can learn to program and become a software developer. They say if you can teach something, it helps you have a better understanding. So hopefully this blog will be useful to others, as it will be for me.
