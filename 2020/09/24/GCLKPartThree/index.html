Google CodeLabs Kotlin Part Three

Posted on 2020-09-24 Edited on 2020-09-27 In Google , Kotlin , CodeLabs , Fragments , Navigation Facebook: 0

Google code labs part three covers fragments, defining navigation paths, and lastly starting an external activity. So let’s get right to it and jump into fragments.

What is a fragment? A fragment is a component that holds part of the behaviour and UI of an activity. Fragments are not independent and are connected to an activity. The behaviour is similar to activities however. We can think of fragments like oompa loompas in Charlie and the chocolate factory. Wonka doesn’t need them to make chocolate, but they make the process 10x easier. Some key advantages to using fragments is that they offer modularity, reusability, and lastly adaptability. In essence they are like a sub activity. They have their own lifecycle, and you can add or remove them when an activity is running.

-Navigation components
To first begin using navigation components, you first need to add the dependencies: navigation-fragment-ktx and navigation-ui-ktx in the module level build.gradle file. Then add an ext variable for the navigationVersion in the project level build.gradle file. Navigation destinations are exactly what you would think. They are fragments, activities, or other components that the user navigates to.

-Navigation graph
A navigation graph is where all the possible destinations to go are. To create a navigation graph, you create a new resource file of type Navigation. Usually it is named navigation. You use the navigation graph to define the navigation flow through the app. You make a connection between destinations. The connections are represented as an action, and have an ID.

-Navigation host fragment
Usually it is named NavHostFragment, and acts as the controller for the navigation graph. When the user moves around between the destinations in the navigation graph, the navigation host fragment swaps the fragments and manages the back stack. To decide which fragment is displayed you will need to set an onClick listener for the view.

-The back stack
Think of it like the order of activities you go to when you open an app and interact with it. It is based on the order of each activity that is opened. So when you open an app the title screen gets put on the back stack. Then when you go to a new screen it gets placed on the back stack. If you were to hit the back button, it would then remove the latest activity on the back stack and return to the title screen. It should follow the last in first out principle.

-The back button
The back button is typically at the bottom of your device. It is the button you see in every single app. The button usually takes you to the last screen you were on. There are ways to control where the back button takes a user. In the navigation xml, go to the fragment, and then the attributes pane. Here you can manage the Pop To, and Inclusive behavior. The Pop To setting removes destinations from the back stack, which determines where the user will go when pressing the back button. Selecting the Inclusive checkbox sets the popUpToInclusive attribute to true. All destinations up to and including this destination are removed from the back stack. If an action’s popUpTo attribute is set to the app’s starting destination and popUpToInclusive is set to true, the back button takes the user out of the app. So it is important to know what each attribute does.

-The up button
Screens in an app can have an Up button that appears at the top left of the app bar. The up button should not be shown when the user first enters the app.

-The options menu
The options menu is a menu that the user accesses from the app bar by tapping the icon with the three vertical dots. To create an options menu with a menu item that displays a Fragment. You need to make sure the Fragment has an ID. Then define the options menu and code the onOptionsItemSelected() handler for the menu items.

-The navigation drawer
The navigation drawer is a panel that slides out from the edge of the screen. Usually it looks like a tab that can have a drop down menu of all the activities you can go to.

-PopTo Not-Inclusive
Pops off everything until it finds the referenced fragment

-PopTo Inclusive
Pops off everything on the back stack including the referenced fragment

-Safe args and Bundles
Safe args is a plugin that you add to your gradle file. It helps prevent bugs in regards to passing an instance of the Bundle class between fragments. A bundle can be thought of like a dictionary. It is a key value pair. Without safe args some of the errors that can occur are type mismatch errors and missing key errors. The safe args generates a NavDirection class for each fragment you have.

-Implicit Intents
An implicit intent declares an action that your app wants to use some other app, such as email.
